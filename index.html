1. Executive Summary

AI에게 마케팅 데이터를 주면 빠르게 분석해 준다. 그러나 '왜'를 모른 채 내리는 판단은, 분석이 아니라 오판의 자동화다. 마케팅 모객 성과가 아닌 인게임 스팟성 세일즈 이벤트로 광고 투자수익률(ROAS)이 급등했을 때 “예산을 증액하세요”라고 권고할 것이고, 플랫폼의 정책 문제로 광고비 지출이 급감했을 때 “매체를 이동하세요”라고 분석할 것이다. 맥락 없는 AI 분석은 속도만 빠른 착각이다.

본 기획서는 AI 에이전트가 매체 API·캘린더 등에서 운영 맥락을 자동 수집하고, 파악되지 않는 맥락만 담당자에게 확인하여 신뢰할 수 있는 의사결정을 돕는 Context-Aware Marketing AI Agent를 제안한다. 나아가 크리에이티브 성과 분석, AI 전략 리포트, 내부 BI 데이터 연동까지 확장하여 반복적 데이터 처리를 자동화하고, 마케터가 전략적 판단에 집중할 수 있는 환경을 만든다.

2. 문제 정의

왜 지금의 AI 마케팅 분석은 위험한가

① LLM은 숫자만 본다 — 맥락 없이 분석하면 잘못된 판단을 내린다
② 검증할 사람이 부족하다 — AI가 틀려도 걸러낼 여력이 없다

AI는 모를 때도 당당하고, 틀릴 때도 유창하다. 오류와 정답이 같은 톤으로 나온다.

가장 위험한 사용자는 AI를 너무 믿는 사람도, 너무 불신하는 사람도 아니다. AI의 자신감에 압도당하는 사람이다. 유창한 분석 결과에 설득당하는 순간, 우리의 판단 능력은 잠시 멈춘다.

2.1 LLM은 '숫자'만 본다

LLM 기반 마케팅 분석의 가장 큰 한계는 단순하다. 데이터는 읽지만, 맥락은 읽지 못한다. 마케팅 데이터의 변동에는 항상 이유가 있다. 인게임 이벤트, 매체 세팅 변경, 플랫폼 정책 이슈, SDK 장애 등 — 숫자 뒤에 숨은 원인을 모른 채 분석하면 필연적으로 분석이 틀릴 수밖에 없다.

다음은 실제 마케팅 현업에서 자주 발생하는 상황을 대입한, 예상되는 오판 시나리오다.

실제 상황LLM의 오판실제 원인
인게임 페이백 이벤트로 ROAS 급등 (18%→42%)"성과 대폭 개선! 예산 증액하세요!"이벤트 종료 시 원상복귀. 증액하면 예산 낭비
구글 정책 위반으로 광고 블락 → spend 급감"Google 성과 급감! Meta로 예산 이동 권고"정책 이슈. 이의제기하면 복구됨
소재 교체 (기존 OFF + 신규 등록) 또는 이벤트 종료로 소재 OFF"캠페인 효율 급락! CPI 상승, 기존 소재 복원 권고"의도적 교체/종료. ML 재학습 기간의 일시적 변동
AppsFlyer SDK 장애로 iOS 이벤트 수집 지연"iOS 설치 급감! CPI 폭등!"SDK 지연으로 인한 일시적 집계 오류

이 모든 오판의 공통점은 하나다. 데이터에는 '무엇이' 변했는지만 있고, '왜' 변했는지는 없다. LLM에게 숫자만 주면 틀린다. 숫자와 함께 맥락(Context)을 줘야 정확한 판단이 가능하다.

2.2 데이터는 넘친다. 그러나 담당자는 한두 명이다

AppsFlyer, 매체 대시보드, 내부 BI 시스템 — 마케팅 데이터는 넘친다. 그러나 이 데이터를 추출하고, 피벗하고, 해석하여 서비스 지표와 연계된 인사이트를 뽑아내는 업무는 1~2명의 담당자에게 집중되어 있다. 데이터 소스가 늘어날수록 분석 업무량은 커지지만, 이를 처리할 인력은 한정되어 있다. 업무가 밀릴수록 분석의 깊이는 얕아지고, 산출물의 퀄리티는 낮아질 수밖에 없다.

상황결과
팀장이 "어제 성과 어때?" 물어봄담당자가 데이터를 뽑아 정리할 때까지 대기
유관부서에서 캠페인 현황 요청리포트 완성까지 현황 파악 불가
담당자 부재 (휴가, 미팅)백업 담당자가 대응하거나, 업무 우선순위 밀림
긴급한 성과 이상 발생담당자가 직접 확인하기 전까지 원인 파악 불가

이것은 능력의 문제가 아니라, 구조상 한계다. 한정된 인력이 반복적인 데이터 정리와 리포팅을 진행하는 구조에서, AI가 내놓는 분석 결과가 늘어날수록 검증할 여력은 자연히 줄어든다. 2.1의 잘못된 분석이 걸러지지 않고 그대로 의사결정에 반영될 수 있다.

AI만 믿으면 판단을 그르치고, 사람에게만 의존하면 한계가 있다.
AI가 맥락을 이해하되, 최종 판단은 사람이 하는 구조가 필요하다.
AI는 '분석'까지. '의사결정과 책임'은 사람이.

3.1 맥락 수집 시스템 (Context Layer)

LLM이 판단을 그르치지 않으려면 맥락이 필요하다. 그러나 담당자가 매번 직접 기록하면 또 다른 업무 부담이 생긴다.

그래서 매체 API·캘린더·장애 로그 등에서 자동 수집 가능한 맥락은 시스템이 먼저 수집하고, 자동으로 파악되지 않는 맥락(변경 이유, 예상치 못한 외부 요인 등)만 담당자에게 확인한다. 담당자의 입력 부담을 최소화하면서도 맥락의 완결성을 확보하는 구조다.

Layer 3 — 성과 데이터 (API 연동)

AppsFlyer MCP, 매체 대시보드, 내부 BI에서 실시간으로 수집되는 성과 데이터다. Layer 1의 도메인 지식과 Layer 2의 맥락이 이 데이터와 결합될 때 비로소 신뢰할 수 있는 분석이 가능해진다.

🌱 같은 데이터, 다른 판단

3개 Layer가 결합했을 때 나타나는 차이다. 앞서 본 페이백 이벤트 케이스를 다시 보자. 같은 숫자를 보고도, 맥락의 유무에 따라 결론은 완전히 달라진다.

3.2 AI 판단 검증 장치 (가드레일)

에이전트에게 맥락과 도메인 지식을 부여하는 것만으로는 충분하지 않다. AI의 판단에 ‘자신감 수준’을 강제로 표기하여, 사람이 검증 없이 수용하는 것을 구조적으로 방지해야 한다. 이를 위해 3중 안전장치를 설계한다.

3.2.1 판단 신뢰도 등급 시스템

에이전트의 모든 판단에 신뢰도 등급을 강제로 부여한다. 유창한 답변의 자신감에 속지 않도록.

3.2.2 의사결정 체크리스트 (Decision Guard)

에이전트가 예산 증감, 매체 ON/OFF 같은 고위험 권고를 생성할 때 반드시 거치는 6항목 체크리스트다.

  • 데이터 충분성 — 최근 7일 이상의 데이터에 기반하는가? (1-2일 데이터로 판단 금지)
  • 이벤트 분리 — 분석 기간에 프로모션·이벤트 기간이 포함되어 있지 않은가?
  • ML 안정화 — 매체 설정 변경 후 ML 재학습 기간(7일)이 경과했는가?
  • 외부 영향 배제 — 정책 변경·SDK 장애 등 외부 요인이 데이터에 영향을 미치지 않았는가?
  • 이상치 배제 — 이상치 탐지(Layer 2-④)에서 플래그된 날짜가 분석 기간에 포함되어 있지 않은가?
  • 크로스 매체 일관성 — 같은 기간 다른 매체와 비교했을 때 일관된 트렌드인가?

출력: 리포트 내 권고 항목에 ✅ PASS 또는 ❌ FAIL 배지를 항목별로 표기한다.

하나라도 FAIL이면: 해당 권고는 '⚠️ 의사결정 보류 — 추가 데이터 필요' 상태로 출력되며, 예산 증감·매체 ON/OFF를 권고하지 않는다.

Context 미응답 시 Fallback: 담당자가 24시간 내 미응답 → 해당 기간 신뢰도 자동 1단계 하향 + 리포트에 ⚠️ 맥락 미확인 배지 부착. 48시간 초과 시 리마인더 자동 재발송.

4.1 크리에이티브 인텔리전스

ⓘ 기술 제약: 현재 MMP·매체 API는 소재 이미지/영상 원본을 반환하지 않으며, 메타데이터(소재명·유형·포맷)와 성과 수치만 제공한다. 따라서 소재 비주얼 자체의 분석(위닝 포인트, 톤앤매너 등)은 현 API 구조상 불가하다.

4.2 AI가 제공하는 풀스택 리포트

수동 리포트 작성을 자동화한다. Section 3의 맥락 기반 분석과 4.1의 크리에이티브 분석을 결합하여, 3종 리포트를 자동 생성한다.

캠페인 성과 리포트

What Happened — 무슨 일이 있었는가

매체별 성과를 맥락과 함께 요약하여, 수치 변동의 원인을 빠르게 파악할 수 있도록 돕는다.

제공 정보매체별 KPI 요약, 이상치 원인 설명, 기간별 트렌드 비교, Context Layer 연동 맥락
활용일간/주간 성과 점검, 이해관계자 보고, 이상 징후 원인 파악

크리에이티브 리포트

What Works — 어떤 소재가 효과적인가

소재 단위의 성과를 정량화하여, 효율 소재 패턴을 도출하고 비효율 소재를 식별한다.

제공 정보소재별 성과 등급(S/A/B/C), Top 퍼포먼스 유형·네이밍 패턴, 저효율 소재 Off 제안
활용소재 운영 의사결정, 다음 소재 기획 방향 참고, 크리에이티브팀 공유

전략 리포트

What To Do — 다음에 무엇을 해야 하는가

성과 데이터와 맥락을 기반으로 다음 주기의 액션 아이템을 도출한다.

제공 정보액션 아이템 목록, 예산 배분 조정 제안, 매체 믹스 변경 제안, 리스크 알림(신뢰도 포함)
활용주간 운영 계획 수립, 예산 의사결정 참고, 리스크 사전 대응

3종 리포트가 What Happened → What Works → What To Do를 순서대로 커버한다. 담당자는 리포트를 검토하고 의사결정에 집중하면 된다.

5. 기존 도구와의 차별점

MMP, 범용 AI와 근본적으로 다른 접근

AppsFlyer (MMP) 일반 AI (ChatGPT 등) 우리의 Context Agent
맥락 수집 없음 (데이터만 제공) 사용자가 직접 입력해야 함 API 자동 수집 + 미확인 맥락만 능동 질문
데이터 분석 대시보드로 수치 보여줌 데이터를 주면 텍스트 분석 제공 데이터 + 맥락을 함께 분석
이상 탐지 Protect360 (프로드=가짜설치) 탐지 기능 없음 운영 이상 탐지 + 즉시 담당자 질문
원인 분석 원인 분석 없음 (WHAT만 보여줌) 운영 맥락 모름 → 할루시네이션 수집 맥락 + WHY 후보 자동 매칭
크리에이티브 대시보드에서 수치 나열 소재 데이터 주면 일반적 분석 소재 On/Off 제안 + 퍼포먼스 패턴 요약
리포트 생성 사람이 직접 수동 작성 What Happened 리포트만 Why + What To Do 전략 리포트

핵심 차별점: 맥락(Context) 기반 원인 분석 + 크리에이티브 인텔리전스 + 전략 리포트 자동 생성. 기존 도구는 'WHAT'을 보여주고, 우리의 에이전트는 'WHY'와 'WHAT TO DO'를 제시한다.

6. 기술 구조: Context-Aware Agent 아키텍처

Context-Aware Marketing AI Agent는 아래와 같은 NHN 인프라 위에서도 구축 가능할 것으로 판단된다.

사용 시나리오

  1. 오전 9시, 출근. Dooray 메신저에 오늘 새벽 에이전트가 생성한 Daily 모객 현황 리포트가 도착해 있다. 열어보면 매체별 모객 성과와 최근 추이가 한눈에 정리되어 있다.
  2. 숫자가 좋지 않다. Google CPI가 전일 대비 8% 올랐고, 리포트에도 “주의” 표시가 붙어 있다. Dooray 메신저에 “Google CPI 급등 원인 분석해줘”라고 물으면, 에이전트가 매체 데이터와 최근 운영 이력을 교차 분석해 추정 원인을 정리해 준다.
  3. 그런데 에이전트가 모르는 맥락이 있다. 어제 오후 서비스 긴급점검이 있었지만, 이 정보는 캘린더에 반영되지 않았었다. 담당자가 “어제 12시~18시 서비스 긴급점검 진행”이라고 알려주면, 에이전트는 해당 맥락을 Dooray 캘린더와 위키에 기록하고 분석을 보정한다.
  4. 다음 날부터는 알아서 반영된다. 한 번 전달한 맥락은 위키에 축적되어, 이후 리포트와 분석에 자동으로 적용된다.

9. 우리의 접근

AI에게 무엇을 맡기고, 무엇을 쥐고 있을 것인가

이 경계를 아는 것이 AI를 잘 쓰는 것이고, 우리 시스템은 그 경계를 구조로 만든다.

INSIGHT 1 — AI를 잘 쓴다는 것

“잘 쓴다”는 건 많이 쓰는 것도, 빠르게 쓰는 것도 아니다.
무엇을 맡기고, 무엇을 내 손에 쥐고 있어야 하는지 아는 것이다.

→ 우리의 접근: Human-in-the-Loop. 의사결정은 반드시 사람이 확인한다. 이 경계를 시스템으로 구현한 것이 가드레일이다.

INSIGHT 2 — 블랙박스 시대에 남는 것

에이전트가 데이터를 처리한다. 하지만 어떤 데이터가 중요한지는 사람이 안다.
남는 건 목표를 정의하는 사람, 맥락을 아는 사람이다.

→ 우리의 접근: Context Layer. 현장 전문가의 맥락 지식을 시스템에 녹여 AI가 올바른 판단을 할 수 있는 토대를 만든다.

INSIGHT 3 — AI는 사고를 증폭하는 파트너

AI는 정답을 주는 기계가 아니라, 사고를 증폭하는 파트너다.
맥락(Context)이 영혼을 만든다.

→ 우리의 접근: 같은 ROAS 42%도 “페이백 이벤트 기간”이라는 맥락이 붙는 순간 완전히 다른 의미가 된다. 데이터를 지식에서 설득으로 전환한다.

본 프로젝트는 이 세 가지 통찰을 하나의 파이프라인으로 구현한다.
